Explorer · post-mortem on a corrupted log

A weir daemon was killed mid-write last night — maybe an OOM, maybe a yanked power cable. Before you restart it, you need to know what's actually on disk: which segments are intact, which are damaged, and what's still recoverable. The Explorer opens the write-ahead-buffer read-only, so you can do exactly that — even against a stopped daemon.

read-only works on a stopped daemon corruption is shown, not crashed-on
The scenario

weir is a durable buffer: producers get a fast ack the moment a record is fsynced to a local WAB segment, and an async drain later forwards sealed segments downstream. After an unclean shutdown, the WAB directory holds the ground truth of what was accepted. This view is how you read it — no daemon required, nothing written back.

The tree below is loaded with a sample post-crash directory: a clean drained segment, a byte-flipped one, a half-sealed one, and a segment still being written.

What you're looking at

State chip on each segment — active (still being written), sealed (full, awaiting drain), or confirmed (already drained downstream, safe to delete).

Integrity badge on sealed segments — ✓ verified, or ✗ + the failure kind. The sample has two casualties: seg_00000002 shows ✗ CrcMismatch (a flipped byte — the stored file CRC no longer matches), and seg_00000004 shows ✗ MissingFooter (the daemon died before the seal finished — a torn tail).

Records (right pane, after you click) — index · length · per-record CRC ✓/✗ · payload preview with a hex⇄utf8 toggle. A damaged record renders as an explicit error row exactly where the corruption starts, and a clean-end vs torn-tail marker tells you whether the segment terminated cleanly.

Why it matters

This is forensic, read-only inspection: point it at a stopped daemon's directory and see precisely what survived the crash, which segment is poison, and which records are recoverable — before you touch anything. It reads through the same weir-wab API the daemon itself uses, so what you see is the on-disk reality, not a guess.

The real tool

This page is a sample-data mock (no backend). Run the real thing against your own WAB directory:

cargo run --manifest-path tools/weir-console/Cargo.toml -- --wab-dir <dir>
# opens the Explorer at http://127.0.0.1:8787 — read-only, localhost only

weir-console — one local tool, three views (Explorer · Ops · Live). The Explorer is strictly read-only; Ops and Live act on a running daemon. Full reference at the docs site.
