The crates

weir isn't only a daemon you run — it's a workspace of crates you can compose. Each piece is usable without the daemon, so you take just the parts you need.

The boundaries are real — enforced by the dependency graph, not convention. weir-core is the only shared base; the libs above it never pull the daemon.

The crates

weir-client

libUnix-only clientno async runtime

Synchronous producer client. One blocking socket round-trip per push(); Durability/NackReason re-exported so a basic producer needs only this one crate.

Reach for it when you produce records from your own app. is_poisoned() / is_recoverable() drive reconnect from a long-lived or async-bridged producer. The client type is #[cfg(unix)]; on Windows only the re-exports are available.

→ docs

weir-sink-sdk

libdeps: weir-core onlyno daemon

The Sink trait plus its SinkError / CommitResult / SinkHealth contract — the stable, frozen interface for forwarding records downstream.

Reach for it when you forward to a custom destination. You can author and fully unit-test a sink with no daemon and no async runtime (a no-op-waker block_on drives commit). Running it in the shipped daemon is a small fork today — a first-class entry point is a future minor.

→ docs

weir-wab

libCross-platformdeps: weir-core + crc32fast

Standalone, runtime-free reader of the on-disk WAB segment format. Point SegmentReader at any segment file and it streams each record, verifying its CRC32. FORMAT_VERSION = 1 is frozen.

Reach for it when you build an offline recovery / export / inspection tool. This is the same parser the daemon and weir-ctl use — one implementation, no drift.

→ docs

weir-core

libCross-platformfrozen API

The wire-protocol types every other crate shares: Envelope, Header, Durability, NackReason, Payload, MAX_PAYLOAD_HARD_CAP. Tiny dep tree (bytes + crc32fast).

Reach for it when you need the low-level wire types directly — most consumers get them transitively via weir-client / weir-wab.

→ docs

weir-server

bin + libUnix only

The daemon: socket layer, WAB, bounded queue, worker pool, drain, metrics, config. Built-in sinks: noop, http, mysql, postgres, and clickhouse (opt-in feature). TCP + mutual TLS behind the tls feature.

Reach for it when you run the buffer as a service. Depend on it only when you actually run the daemon — everything else is in the smaller crates.

→ docs

weir-ctl

binno weir-server dep

Admin & inspection CLI: health, push, metrics, segments (per-shard WAB inspect), and dl (dead-letter list / drop / requeue). A thin tool over the socket + the WAB reader.

Reach for it when you operate a running daemon. Installing it doesn't drag in the daemon's dependencies. (Live commands use --socket/--addr; segments/dl read the WAB on disk via --wab-dir.)

→ docs

weir-testkit is an internal test harness (publish = false) — not part of the public API and not for downstream use.
