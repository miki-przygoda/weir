# Fuzzing

weir's on-disk and on-wire byte parsers sit on trust boundaries: the
WAB confirmed-file format is read at daemon startup (post-restart the
bytes are attacker-controlled if the host was compromised), and the
wire envelope decoder reads every byte off an unauthenticated socket
connection. A panic or unbounded allocation in either parser turns
into a denial-of-service vector.

This directory ships a `cargo-fuzz` setup with two targets:

| Target | Function under test | Why fuzz it |
| --- | --- | --- |
| `wab_confirmed` | `weir_server::wab::format::parse_confirmed` | Parses the on-disk `.confirmed` sidecar. Reached at startup by the drain's directory scan and at runtime when the drain writes confirmation records. |
| `envelope_parse` | `weir_core::Envelope::decode` | Decodes the wire-format envelope from every push the daemon accepts. Every connected client gets to feed this function arbitrary bytes. |

The existing proptest harness in `crates/weir-core/tests/reference_frames.rs`
covers *round-tripping* — encode → decode of valid envelopes always
yields the original. The fuzz targets cover the *inverse*: arbitrary
input never panics, regardless of whether it round-trips.

## Running

`cargo-fuzz` requires a nightly Rust toolchain (libFuzzer's sanitizer
plumbing isn't on stable):

```sh
rustup install nightly
cargo install cargo-fuzz
```

Then, from the repo root:

```sh
# 60-second smoke run of one target:
cd fuzz
cargo +nightly fuzz run wab_confirmed -- -max_total_time=60

# Longer runs surface deeper coverage:
cargo +nightly fuzz run envelope_parse -- -max_total_time=600

# All targets, sequentially, 60 s each:
for t in $(cargo +nightly fuzz list); do
    cargo +nightly fuzz run "$t" -- -max_total_time=60
done
```

`cargo fuzz` builds a release-with-sanitizer binary, then invokes
libFuzzer's coverage-guided loop. On a panic or sanitizer hit it
writes the offending input to `fuzz/artifacts/<target>/`. Reproduce
locally with `cargo +nightly fuzz run <target> fuzz/artifacts/<target>/<crash-file>`.

## Seed corpus

For coverage-guided fuzzing, libFuzzer benefits from a seed corpus of
"interesting" inputs to start from. Two natural sources for these
parsers:

- **`weir_confirmed`**: the bytes written by
  `weir_server::wab::format::build_confirmed`. Seed inputs can be
  generated by running the existing format tests in
  `crates/weir-server/src/wab/format.rs::tests` and dumping each
  generated buffer to `fuzz/corpus/wab_confirmed/`.
- **`envelope_parse`**: the reference frames in
  `crates/weir-core/tests/reference_frames.rs` are
  already-bytes-on-disk and load straight into
  `fuzz/corpus/envelope_parse/`.

The corpus directories are gitignored — each invocation of
`cargo fuzz run` re-discovers and grows them.

## CI integration

Not wired up. cargo-fuzz needs nightly Rust which would make
GitHub Actions jobs slower to start and adds a different toolchain
to the CI matrix. A reasonable starter shape — a 60-second smoke fuzz
on each PR — would live in `.github/workflows/fuzz-smoke.yml` and is
a clean follow-up when someone wants it.

## What's deliberately NOT fuzzed (yet)

- **`SegmentReader::open + iterate`** — the per-record loop in
  `crates/weir-server/src/wab/mod.rs`. Reaches MAX_PAYLOAD_HARD_CAP
  before allocating, so the worst case is "drop an error"; we'd
  still like coverage. Adding the target requires either exposing
  `SegmentReader` from the `weir-server` library facade (which
  re-introduces visibility-leak warnings because `Metrics` is
  `pub(crate)`) or extracting the parsing into a pure-bytes function
  in `wab/format.rs`. The pure-bytes extraction is the cleaner path
  and is a one-PR follow-up.
- **`weir_server::config::Config` TOML parsing** — exposed surface is
  small (operator-controlled, runs once at startup), so the
  ROI of fuzzing it is low.
