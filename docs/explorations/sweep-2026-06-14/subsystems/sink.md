# Sink subsystem — adversarial verification (sweep 2026-06-14)

Audited 11 findings across `mysql.rs`, `postgres.rs`, `clickhouse.rs`, `http.rs`, against the drain caller and config defaults. 9 confirmed real, 1 refuted, 1 partially refuted (severity downgraded). Crown invariant (no false acks) holds throughout — all confirmed findings are correctness-adjacent (data routed to dead-letter / duplicate inserts / test gaps), not false-ack data loss.

## Confirmed (real)

### MySQL sink dead-letters live data on a recoverable connection-time auth/access failure (asymmetric with Postgres)
- `crates/weir-server/src/sink/mysql.rs:268` — severity **high**
- Argument: `commit()` does `self.pool.get_conn().await.map_err(classify)` (mysql.rs:268). `classify()` routes `E::Server(srv)` through `is_transient_server_code(srv.code)` (mysql.rs:219-226). For ER_ACCESS_DENIED_ERROR (1045) / ER_DBACCESS_DENIED_ERROR (1044) that returns false (explicitly asserted at mysql.rs:487-488), so the error is classified Permanent. A Permanent error returned from `conn_fut` lands in the `Ok(Err(e))` arm of the timeout match (mysql.rs:285); `!e.is_transient()` is true, so the entire batch is dead-lettered (mysql.rs:293-300) and the drain writes it to the dead-letter file (drain/mod.rs:684-696). Postgres deliberately forces every `pool.get()` failure to Transient via `SqlSinkError::transient(DRIVER, format!("pool: {e}"))` (postgres.rs:358-362) — connection establishment is treated as connection-level, never per-batch. The asymmetry is real and the MySQL path diverts live records during a credential-rotation / GRANT-revoke window. The common "server down" case is unaffected (`E::Io` → transient, mysql.rs:213).
- Verdict: **real** — mysql.rs:268 → classify → is_transient_server_code(1045/1044)=false (asserted mysql.rs:487-488) → permanent → dead-letter (mysql.rs:293-300), while postgres.rs:358-362 forces pool errors transient.

### ClickHouse sink commit() request/response classification has no in-process test (only an #[ignore] live-docker integration test)
- `crates/weir-server/src/sink/clickhouse.rs:255` — severity **high**
- Argument: Only the pure helpers are unit-tested (encode_rowbinary, dedup_token, status_is_transient, split_credentials, insert_query — clickhouse.rs:350-538). `commit()` itself — 2xx→committed (clickhouse.rs:296-300), 503/transient→`SqlSinkError::transient` not confirmed (301-305), 4xx→permanent dead-letter (306-313), timeout mapping (280-284), and the `insert_deduplication_token` query param (271-274) — is exercised only by `clickhouse_sink_end_to_end` (system.rs:2096), which is `#[ignore]` + `#[cfg(feature="clickhouse-sink")]` + requires `WEIR_TEST_CLICKHOUSE_URL` (system.rs:2093-2099). The HTTP sink has a full in-process `spawn_mock_server` suite (http.rs:421) covering exactly these mappings, and that helper is directly reusable. A regression flipping transient→permanent (live data to dead-letter), permanent→success (confirm+delete a rejected batch = silent loss), or dropping the dedup token would pass all currently-running tests.
- Verdict: **real** — commit() at clickhouse.rs:255 has zero in-process coverage; its only test is gated by `#[ignore]`+feature+env (system.rs:2093-2099), unlike http.rs's mock-server suite.

### ClickHouse does not percent-decode URL credentials before HTTP basic auth (silent divergence from SQL sinks)
- `crates/weir-server/src/sink/clickhouse.rs:229` — severity **medium**
- Argument: `split_credentials()` slices `creds[..colon]` / `creds[colon+1..]` verbatim out of the URL (clickhouse.rs:241-242) with no percent-decode; `commit()`/`health()` pass them straight to `req.basic_auth(user, Some(pass))` (clickhouse.rs:277, 327). The config shape is documented as a URL `http://[user:password@]host:8123` (clickhouse.rs:89), so an operator with a reserved char in the password naturally percent-encodes (`p%40ss`); ClickHouse then receives the literal `p%40ss`, returns 401, `status_is_transient(401)` is false (clickhouse.rs:220-224, asserted at 511), and the batch is dead-lettered as permanent. The SQL sinks use driver URL parsers (`mysql_async::Opts::from_url`, mysql.rs:149; tokio_postgres Config parse) that percent-decode userinfo per RFC 3986, so the same credential works for SQL sinks but fails for ClickHouse. The existing test `split_credentials_handles_at_sign_in_password` (clickhouse.rs:517) uses a RAW `@` and reinforces non-decoding rather than catching it.
- Verdict: **real** — clickhouse.rs:241-242 slices raw substrings with no percent-decode, then hands them to basic_auth (277); SQL sinks go through RFC-3986 URL parsers, creating the divergence.

### ClickHouse response body read is not bounded by the sink's configured timeout
- `crates/weir-server/src/sink/clickhouse.rs:280` — severity **medium**
- Argument: `timeout` is documented as "per-request timeout" (clickhouse.rs:96-97) but applied only as `tokio::time::timeout(self.config.timeout, req.send())` (clickhouse.rs:280; same in health at 329). The reqwest client is built with no `.timeout()` (clickhouse.rs:195-197), unlike the HTTP sink which sets `.timeout(config.timeout)` on the client (http.rs:114). `req.send()` resolves on response headers, so on the permanent path `resp.text().await` (clickhouse.rs:308) reads the body fully outside any sink-level timeout. A server/proxy that returns headers promptly then stalls the error body hangs commit() until the drain's coarse backstop `commit_timeout` fires (= `sink_timeout_secs*2 min 60s`, main.rs:294; applied at drain/mod.rs:593) — the configured sink_timeout (default 10s, config/mod.rs:529) is silently ignored on that path. Setting `.timeout(config.timeout)` on the client builder closes it.
- Verdict: **real** — client built without `.timeout()` (clickhouse.rs:195-197) while http.rs:114 sets it; body read at clickhouse.rs:308 is outside the `tokio::time::timeout` that wraps only `req.send()` (280).

### HTTP concurrency: no test asserts a transient mid-batch error actually cancels still-in-flight POSTs
- `crates/weir-server/src/sink/http.rs:351` — severity **medium**
- Argument: The comment at http.rs:351-357 claims dropping the buffered stream on a transient error "cancels any still-in-flight POSTs"; this is load-bearing for the at-least-once double-delivery bound. The two new concurrency tests only assert order preservation (http.rs:537) and overlap (http.rs:552). `transient_in_middle_of_batch_returns_err` (http.rs:699) uses 2 records and asserts only `err.is_transient()` (http.rs:710) — it never counts requests that reached vs. were dropped, so the cancellation-on-drop behaviour of `futures_util::buffered` is asserted nowhere.
- Verdict: **real** — http.rs:699-711 asserts only the Err; no test counts server-side requests to prove in-flight POSTs are cancelled.

### HTTP concurrency: no test pins record-to-outcome pairing under concurrent POSTs
- `crates/weir-server/src/sink/http.rs:684` — severity **medium**
- Argument: Each record is paired with its own outcome via `.map(|record| async move { ...(record, outcome) })` (http.rs:326-331). `mixed_batch_committed_and_dead_lettered` (http.rs:684) sends 3 records at concurrency 8 but asserts only counts (`committed.len()==2`, `dead_lettered.len()==1`, http.rs:695-696) — never which payload landed where. The only content assertion on a dead-letter reason (`http_400_dead_letters`, http.rs:616-618) uses a single-record batch and cannot catch a concurrency-only mis-pairing. The comments at http.rs:685-686 ("second 400", "Order of records … determines which gets which response") are stale: with concurrency 8 all three POSTs are issued concurrently, so which canned response a record receives is non-deterministic; the test passes only because assertions are count-based. A regression to `buffer_unordered` or a zip-based pairing could mis-attach a dead-letter reason to the wrong payload undetected.
- Verdict: **real** — http.rs:695-696 asserts counts only; stale comment at 685-686 contradicts the concurrency=8 model; no content/identity assertion exists for the concurrent path.

### ClickHouse content-derived dedup token silently breaks idempotency if sink_max_batch_size changes across a restart
- `crates/weir-server/src/sink/clickhouse.rs:72` — severity **low**
- Argument: `dedup_token()` hashes the exact sub-batch handed to commit() (clickhouse.rs:72-84). The drain re-splits each segment into sub-batches of `sink.max_batch_size()` (drain/mod.rs:505,527), which is `sink_max_batch_size`. The token is the only crash-replay dedup mechanism for ClickHouse (clickhouse.rs:12-19). Within a run the split is deterministic so tokens are stable. But restarting with a different `sink_max_batch_size` before an unconfirmed segment is replayed regroups records into different sub-batch boundaries → different tokens → ClickHouse fails to recognise the re-inserted block as a duplicate → duplicate rows. Surfaced in no operator-facing place; should at minimum be documented (don't change `sink_max_batch_size` while unconfirmed segments exist) or, better, derive the token from per-segment identity.
- Verdict: **real** — token derived from the mutable sub-batch grouping (clickhouse.rs:72-84) that the drain re-splits by `max_batch_size()` (drain/mod.rs:505,527); a changed batch size across restart yields different tokens.

### ClickHouse split_credentials leaves username-only userinfo in base_url, contradicting the struct invariant
- `crates/weir-server/src/sink/clickhouse.rs:240` — severity **low**
- Argument: `split_credentials` only enters the strip path when a `:` is found (clickhouse.rs:240 `if let Some(colon) = creds.find(':')`). For `http://user@host:8123` (username, empty password) `creds.find(':')` is None, so it falls through to `(url.to_string(), None)` (clickhouse.rs:248): base_url retains `user@` and credentials is None. This contradicts the documented invariant on `base_url` ("any user:password@ userinfo stripped; credentials are applied per-request as basic auth", clickhouse.rs:142-145). reqwest still applies the embedded username so it functionally connects, but the userinfo is carried verbatim into every POST/GET URL (clickhouse.rs:270, 324) rather than via the basic_auth path the rest of the code assumes. Fix: strip whenever `@` is present in the authority (the `at` is already located at clickhouse.rs:238), capturing an empty password.
- Verdict: **real** — the `rfind('@')` block at clickhouse.rs:238 only strips inside the `find(':')` branch (240); a username-only authority falls through to (url, None) at 248, leaving `user@` in base_url against the 142-145 invariant.

### ClickHouse RowBinary encoding silently assumes a plain String column with no config validation
- `crates/weir-server/src/sink/clickhouse.rs:52` — severity **low**
- Argument: `encode_rowbinary()` emits `leb128(len) ++ bytes` (clickhouse.rs:52-59), the RowBinary wire form for exactly one ClickHouse `String` column — stated only in the private fn's comment (clickhouse.rs:49-50). RowBinary is positional/untyped on the wire; for FixedString(N) / LowCardinality(String) / Nullable(String) the length prefix is misparsed and the insert is corrupted or rejected with no validation. `ClickHouseSinkConfig` (clickhouse.rs:88-98) and the module docs (clickhouse.rs:1-19) never state the column must be plain `String`, unlike the MySQL/Postgres module docs that spell out the column requirement (VARBINARY/BLOB at mysql.rs:10-16; BYTEA at postgres.rs:17-26).
- Verdict: **real** — the String-column contract lives only in a private fn comment (clickhouse.rs:49-50); module docs (1-19) and config (88-98) omit it, unlike mysql.rs:10-16 / postgres.rs:17-26.

## Refuted / dismissed

### Concurrent HTTP sink discards already-accumulated dead_lettered records when a later record hits a transient error
- `crates/weir-server/src/sink/http.rs:351` — claimed severity **low** → verdict **refuted**
- Argument as claimed: on the first transient outcome, `return Err(e)` (http.rs:357) drops the partially-built `dead_lettered` vec; on retry the permanently-rejected records re-POST and can be dead-lettered multiple times, and a batch that reliably mixes a 400 and a 503 makes no forward progress.
- Why refuted as a defect: this is the explicitly documented and intended contract, not a regression. The comment at http.rs:351-357 states a transient error aborts the batch so the drain retries the whole segment, and the per-record idempotency key covers re-sends. At-least-once is preserved (the claim concedes nothing is falsely confirmed). The "serial design didn't have this" framing is incorrect: a serial per-record loop returning Err on the first transient error would equally discard earlier dead-letter accumulation — the behaviour is identical, the concurrency change did not introduce it. The pathological "always 400 + always 503 in the same batch" case is a genuinely stuck endpoint that the drain's MAX_RETRIES backstop bounds; treating a mixed batch differently (the suggested "drain to completion") is a design preference, not a correctness fix. No defect in code.
- Verdict: **refuted** — http.rs:351-358 implements the documented "transient aborts the segment" contract; at-least-once holds, and a serial loop would behave identically, so this is intended behaviour, not a regression or bug.

### MySQL transient code 1317 (ER_QUERY_INTERRUPTED) is undocumented and untested
- `crates/weir-server/src/sink/mysql.rs:242` — claimed severity **low** → verdict **uncertain** (partially refuted; downgrade to **info**)
- Argument as claimed: `is_transient_server_code()` matches 1317 (mysql.rs:242) but the module docstring (mysql.rs:30-36) and `is_transient_server_code_table` test (mysql.rs:476-490) omit it; an operator reading the module docs won't learn 1317 is retried and a future edit could silently flip it.
- What holds: the module docstring (mysql.rs:27-39) enumerates 1205, 1213, 1290, 1062 and omits 1317; the test (mysql.rs:476-490) asserts 1205/1213/1290/1062 transient and several permanent codes but never asserts 1317. Both gaps are real as stated.
- What refutes the framing: 1317 IS documented operator-facing in `docs/operations/configuration.md:718` (`| 1317 ER_QUERY_INTERRUPTED | transient | retry |`). The claim's "undocumented" is true only of the in-module docstring, not the operator documentation an operator would actually consult. The test-coverage gap is genuine but it is a missing one-line assertion on a pure, conservatively-short matcher, not a defect in behaviour. Severity is over-stated at low; this is an info-level doc/test consistency nit.
- Verdict: **uncertain** — 1317 is genuinely absent from the module docstring (mysql.rs:27-39) and the test (mysql.rs:476-490), but it IS documented at docs/operations/configuration.md:718, so the "undocumented" claim is only partly true and the impact is info-level, not low.
