# Socket subsystem audit — sweep 2026-06-14

All six reported findings were verified against the code and confirmed **real**; severities largely hold, with two semantic/robustness items that are genuine but narrow.

## Confirmed (real)

### 1. UnknownMessageType/UnknownDurability decode errors nacked as InternalError (documented open) but connection is closed
- **File:** `crates/weir-server/src/socket/connection.rs:414` (and the close at `:150`)
- **Severity:** medium
- **Argument:** `Header::decode` (`crates/weir-core/src/envelope.rs:165-201`) checks magic (167) → version (171) → header CRC (180) *before* parsing the typed fields, so the `UnknownMessageType` (envelope.rs:188) and `UnknownDurability` (envelope.rs:190) errors are raised on a frame whose header CRC is valid: the bad enum byte is uncorrupted, i.e. a genuine permanent client protocol error. `nack_for_decode_error` (connection.rs:401-416) maps both through the `_ =>` arm at line 414 to `WireNack::InternalError` + `MetricNack::internal_error`. The decode-error site then does `send_nack(...); return Ok(())` (connection.rs:141-150), closing the connection. `docs/wire_protocol.md:147` documents `Push → Nack(InternalError)` as "open — queue saturation, transient", the single Nack for which the server keeps the connection open. Consequences confirmed: (a) the client sees a Nack whose documented contract is "open/transient/retry" yet the connection is closed; (b) it is indistinguishable from a real pipeline `InternalError` (handle_push lines 333/355/371 emit the same wire+metric pair), so a retry-on-InternalError client loops against a frame that can never succeed; (c) `records_nack{reason=internal_error}` is polluted by malformed client input. The N-series added `EmptyPayload` (nack.rs:29; connection.rs:172-190) as a distinct reason but left these two enum-decode errors collapsed into InternalError; `NackReason` defines only 0x01..0x07 with no code for unknown type/durability, so a full wire-clean fix needs a future wire version.
- **Verdict:** real — UnknownMessageType/UnknownDurability arise post-CRC (envelope.rs:188/190), hit the InternalError `_` arm (connection.rs:414), and the path closes the connection at connection.rs:150, contradicting docs/wire_protocol.md:147.

### 2. Payload/CRC reads are not raced against handler_shutdown; mid-frame stall holds a permit until read_timeout during graceful shutdown
- **File:** `crates/weir-server/src/socket/connection.rs:195` (and `:213`)
- **Severity:** low
- **Argument:** The header read uses `tokio::select! { biased; res = timeout(read_timeout, read_exact(header)), _ = shutdown_rx.changed() => return Ok(()) }` (connection.rs:107-114), so an idle connection exits promptly on shutdown. The payload read (line 195) and CRC read (line 213) are plain `timeout(config.read_timeout, read_exact).await` with no shutdown race. Once a header has been consumed, a client that stalls mid-frame at shutdown is not woken by `handler_shutdown_tx.send(true)` (mod.rs:226) and sits until read_timeout. Defaults confirmed (config/mod.rs:469/482; connection.rs:35): shutdown_timeout_secs = read_timeout = ACK_TIMEOUT = 30, so such handlers can consume the entire drain window and tip the daemon into the `abort_all` fallback (mod.rs:235-249), contradicting the module's stated intent that idle handlers "exit immediately on the next loop check" (mod.rs:229). No data loss (at-least-once + abort_all backstop) — an availability/clean-shutdown gap and an asymmetry with the header path. The TCP+mTLS handler shares the same body (tcp.rs:222) and inherits the gap.
- **Verdict:** real — connection.rs:195/213 wrap read_exact in a bare timeout with no `shutdown_rx.changed()` arm, unlike the header read at 107-114.

### 3. Payload/CRC read timeout is a whole-transfer deadline but documented and metered as a per-byte idle timeout
- **File:** `crates/weir-server/src/socket/connection.rs:195` (and `:213`)
- **Severity:** low
- **Argument:** `tokio::time::timeout(config.read_timeout, stream.read_exact(...))` makes `read_timeout` a total deadline for the whole payload (and CRC) phase, not an idle/per-byte timeout. But `ConnectionConfig::read_timeout` is documented as "Maximum time the handler will sit in `read_exact` waiting for the next byte" (connection.rs:43-47) and the `weir_connection_idle_timeout` metric is described as connections that sat in read_exact "without receiving the next byte" (metrics/mod.rs:322-323). A legitimate client streaming a large payload (up to MAX_PAYLOAD_HARD_CAP = 16 MiB) over a slow-but-steady link is killed at read_timeout (30s default → needs sustained ~560 KB/s) even though it never went idle, and `connection_idle_timeout` is incremented for it (connection.rs:199/217) — a non-idle condition mislabeled in the metric. Impact is narrow (slow links + large payloads); the security direction is fine (a slowloris cannot drip-feed to evade the cap). The header phase (16 bytes) shares the same total-deadline semantics but is immaterial at that size.
- **Verdict:** real — the timeout wraps the entire read_exact (whole-transfer), while connection.rs:43-47 and metrics/mod.rs:322-323 both describe per-byte idle semantics.

### 4. Shutdown-time socket removal is path-based (std::fs::remove_file), inconsistent with the unlinkat-based startup cleanup
- **File:** `crates/weir-server/src/socket/mod.rs:253`
- **Severity:** low
- **Argument:** `run()` ends with `let _ = std::fs::remove_file(&config.socket_path)` (mod.rs:253) — a by-name unlink following the final path component as it exists at shutdown. This is the exact TOCTOU surface the startup bind sequence was hardened against: `bind_hardened`/`cleanup_existing_socket` use a pinned dirfd + `unlinkat(dirfd, basename, ...)` + AT_SYMLINK_NOFOLLOW + inode-equality checks (mod.rs:343-415, 484-530), and `docs/security/socket-bind.md:85,109-111` explicitly switches step 4 from `remove_file` to `unlinkat` for this reason. Under a group/world-writable parent (the case `warn_if_parent_world_writable` warns about, e.g. /tmp; mod.rs:550-584), an attacker who swaps a different inode in at the daemon's path before shutdown gets that inode unlinked by the daemon. The `listener` is still held in scope until `run` returns, so the bound socket inode is already detached from new clients; removal via the retained dirfd, or simply skipping removal, avoids acting on an attacker-controlled name. Impact is bounded (one attacker-chosen unlink, no weir data corruption, no privilege change) and only under an already-flagged hostile parent — hence low.
- **Verdict:** real — mod.rs:253 uses path-based remove_file at shutdown, reverting to the precise pattern the startup hardening (unlinkat, docs/security/socket-bind.md) was built to avoid.

### 5. TLS handshake is not raced against handler_shutdown; in-flight handshakes block graceful drain up to handshake_timeout
- **File:** `crates/weir-server/src/socket/tcp.rs:179`
- **Severity:** low
- **Argument:** In `tcp::run` the spawned task runs `time::timeout(handshake_timeout, acceptor.accept(stream))` (tcp.rs:179-212) before it consults `handler_shutdown`; the watch receiver is only handed to `handle_connection` (tcp.rs:222) after the handshake completes. At shutdown `handler_shutdown_tx.send(true)` fires (tcp.rs:255) but tasks mid-handshake do not observe it; each keeps a semaphore permit and JoinSet slot until handshake completion or `handshake_timeout` (default 10s; config/mod.rs:618). Under a flood of clients that complete the TCP connect then stall the TLS handshake at shutdown, the drain (bounded by shutdown_timeout_secs, default 30s) can be consumed by stalled handshakes and tip connections into `abort_all` (tcp.rs:267-277), inflating `connections_aborted_at_shutdown`. The Unix loop has no analog because `handle_connection` (which holds the shutdown race) runs immediately after accept. Fix: also `select!` the handshake against `handler_shutdown.changed()`.
- **Verdict:** real — the handshake at tcp.rs:179-212 runs before the shutdown-aware handle_connection (tcp.rs:222) with no shutdown race, so handshake_timeout (10s) gates clean drain.

### 6. umask save/restore in bind_hardened has no RAII guard; an unwind between set and restore would leak the tightened 0o177 umask process-wide
- **File:** `crates/weir-server/src/socket/mod.rs:372`
- **Severity:** low
- **Argument:** `bind_hardened` does `let saved_umask = umask_set(0o177)` (mod.rs:372), `let bind_result = UnixListener::bind(path)` (373), `let _ = umask_set(saved_umask)` (374), then `bind_result?` (375). The error `?` is deliberately deferred past the restore, so a bind *error* still restores the umask. Only a panic/unwind in `UnixListener::bind` would skip the restore and leak umask 0o177 process-wide, silently tightening every later file/dir creation. Confirmed this is **latent**: only `UnixListener::bind` runs in the window and it returns `Result` (interior-NUL path yields Err, not panic), so no current unwind path exists — the invariant relies on that call never panicking, which is fragile for security-sensitive umask manipulation. An RAII guard (like `OwnedDirFd`, mod.rs:592-607) would make the restore unwind-safe. The existing test `bind_hardened_restores_umask_after_bind` (mod.rs:785-802) only covers the success path. fix_class refactor / severity low are appropriate.
- **Verdict:** real — restore at mod.rs:374 is a plain statement, not Drop-guarded; correct today only because UnixListener::bind (373) returns Err rather than panicking, a fragile invariant.

## Refuted / dismissed

None. All six findings were confirmed against the code.
