## 🟠 High (2)

### G01 — Sink HTTP/ClickHouse clients follow redirects by default; a redirected POST ending in 2xx is acked as committed though the body was never delivered (false ack)
*(sink · durability/guard · `crates/weir-server/src/sink/http.rs:158`)*

Both sink HTTP clients are built with the reqwest default redirect policy: HttpSink (http.rs:113-124) and ClickHouseSink (clickhouse.rs:211-219) call Client::builder() and never .redirect(...). reqwest 0.12's Policy::default() is Policy::limited(10) (verified in ~/.cargo reqwest-0.12 src/redirect.rs:160-163), so up to 10 redirects are followed automatically; grep confirms no redirect policy anywhere under crates/weir-server/src/sink/. In HttpSink::post_record, any status.is_success() returns Ok(()) (http.rs:157-164), and the record is pushed to `committed` in commit() (http.rs:339). When the configured endpoint (or a fronting proxy/LB) answers the POST with 301/302/303, reqwest re-issues the request as a BODILESS GET to the Location target; if that returns 2xx the sink observes success and marks the record committed even though the payload bytes were never delivered to a real handler. The drain then confirms+deletes the segment on BatchResult::Ok (drain/mod.rs), so the record is gone: a false ack, violating the crown invariant. ClickHouse::commit has the identical exposure — a redirected POST ending in 2xx returns `committed: batch` (clickhouse.rs:350-355) although the INSERT ... FORMAT RowBinary may never have executed. Corroborating evidence that 3xx was assumed observable: health() has a now-dead `|| resp.status().is_redirection()` arm (http.rs:381) that reqwest can never reach because it follows the redirect before returning; and neither commit path classifies 3xx at all (classify_status_transient only matches 408/429/5xx at http.rs:244-251; status_is_transient likewise at clickhouse.rs:242-246; the module doc http.rs:20-27 enumerates 2xx/4xx/transient but is silent on 3xx). 307/308 preserve the body but can redirect to a host the operator never configured (misconfigured or attacker-controlled Location), committing to the wrong destination.

➡️ **Fix:** Set .redirect(reqwest::redirect::Policy::none()) on both sink clients (http.rs:113, clickhouse.rs:211) and explicitly classify 3xx on the commit path as permanent (dead-letter) or transient (retry) — never committed. Disabling redirects also makes the existing health() is_redirection() arm reachable and correct.

### G02 — Transient scan_and_advance_counter() failure leaves next_counter at 1, re-opening the F12 segment-overwrite data-loss hole via a new trigger
*(wab · durability/error-handling · `crates/weir-server/src/wab/mod.rs:439`)*

flusher_thread() calls writer.scan_and_advance_counter() (mod.rs:439) and on ANY error merely logs `failed to scan segment counters; starting at 1` (mod.rs:440) and proceeds. scan_and_advance_counter (segment.rs:435-446) computes max from store.segment_counters(...)? — the `?` short-circuits BEFORE next_counter is touched, so on error next_counter stays at its constructor default of 1 (new_with_store, segment.rs:424). The scan's only fallible step is std::fs::read_dir(dir)? inside FsSegmentStore::segment_counters (segment.rs:358), which fails transiently under fd pressure (EMFILE/ENFILE), backing-fs EIO, or ENOMEM. Crucially, recover_open_segments() runs earlier in spawn() on the calling thread (mod.rs:231) and recover_segment() renames every crashed active .wab to .wab.sealed (recovery.rs:329), so at flusher start a shard dir can legitimately hold ONLY recovered-but-undrained seg_NNNNNNNN.wab.sealed files. If the counter scan then fails, ensure_open() (segment.rs:499-506) opens its first segment as seg_00000001.wab — WabSegment::create uses create_new(true)/O_EXCL (segment.rs:59) which enforces non-existence of the .wab name only and does NOT collide with an existing .wab.sealed — and that segment's first seal does std::fs::rename(seg_00000001.wab -> seg_00000001.wab.sealed) (segment.rs:258), atomically REPLACING the recovered, undrained sealed segment. Those records were acked durable to producers but are lost before the drain delivered them: the exact crown-invariant violation F12 (commit fb02a62) was meant to eliminate. The F12 fix only corrected segment_counter_from_path parsing (file_stem -> leading-digit-run, segment.rs:294-304) so sealed files are counted on the success path; it never touched the mod.rs:439-441 error branch, so the same data-loss outcome stays reachable via this distinct (transient read_dir) trigger. The DST SimSegmentStore::segment_counters delegates straight to FsSegmentStore (dst.rs:427-429) and injects no fault on the scan, so DST does not cover it, and the regression test scan_and_advance_counter_advances_past_sealed_segments (segment.rs:756) asserts only the happy path with .unwrap().

➡️ **Fix:** Do not treat a scan_and_advance_counter() failure as benign. Either (a) propagate the io::Error out of flusher_thread and take the shard permanently offline (Nack-everything, the same end state as the respawn-cap path at mod.rs:166) so nothing is ever sealed at an unestablished counter, or (b) before opening/sealing the first segment, refuse to proceed if any seg_*.wab.sealed exists in the shard dir while the counter could not be established. A cheaper local defense: have ensure_open()/seal() refuse to seal when the target .wab.sealed already exists (turn the implicit overwrite into an explicit error). Add a regression test that injects a segment_counters() read_dir error with a pre-existing seg_00000001.wab.sealed and asserts the recovered sealed segment is never overwritten (the existing scan_and_advance_counter_advances_past_sealed_segments test only covers the happy path with .unwrap()).

### G04 — Opt-in read timeout can desync the connection so a stale response is read as the next push's ack (false ack)
*(client-sdk-ctl · bug/guard · `crates/weir-client/src/unix.rs:154`)*

read_response() (unix.rs:154-189) reads each response in three separate read_exact calls (header at :156, payload at :175, CRC at :179). With the F43 opt-in read timeout set (set_read_timeout, unix.rs:250), a timeout that fires mid-frame returns Err but the bytes already drained from the socket are gone. WeirClient (unix.rs:81-86) holds NO poisoned/closed/desynced flag — only `stream` and `default_durability` — so after the `?`-propagated error the same client is fully reusable. The set_read_timeout doc (unix.rs:244-249) explicitly tells the producer to 'retry' on the Io timeout ('the record may still have been durably written') without warning that the connection MUST be discarded. If the producer retries push() on the SAME WeirClient, the leftover bytes of slow response N are consumed as the response to push N+1: if response N was an Ack but N+1 was actually Nacked (or never processed), push N+1 returns Ok(()) — a FALSE ACK, directly violating the crown invariant. This hazard is NEW: F43 added the timeout, F44 added only the alloc cap; neither addresses reuse-after-timeout, and the only timeout test (read_timeout_bounds_a_silent_daemon_instead_of_blocking_forever, unix.rs:376) does a single push and never reuses the client.

➡️ **Fix:** On any Io error in read_response (especially a timeout), mark the WeirClient poisoned so subsequent push/health_check fail fast, or document loudly that the connection MUST be dropped and re-established after a timeout. Add a test that a second push on a timed-out client does not read a stale frame.

### G05 — weir-ctl `dl drop --yes` bails on the first delete failure, leaving an unreported partial irreversible deletion
*(client-sdk-ctl · error-handling/error-handling · `crates/weir-ctl/src/main.rs:358`)*

cmd_dl_drop iterates segments and `std::fs::remove_file(p).map_err(...)?` (main.rs:358-360) returns on the FIRST failure. If file #k of N fails (EACCES, or the daemon raced and renamed/removed it), files #1..#k-1 are already irreversibly deleted but the only output is `weir-ctl: remove <path>: <err>` (via main.rs:122); the `dropped N segment(s)` summary (main.rs:361-365) never prints, so the operator cannot tell how many irreversible deletions actually happened, and a naive re-run would drop again. For an explicitly-irreversible command, partial progress should be reported.

➡️ **Fix:** Accumulate per-file outcomes: count successes, collect failures, and print 'dropped X of N (Y failed: ...)' rather than bailing on the first error; exit non-zero if any failed.

### G06 — weir-ctl `segments` counts every dead-letter file while `dl list`/`dl drop` apply a `dl_*` suffix filter — the two views can disagree
*(client-sdk-ctl · observability/bug-fix · `crates/weir-ctl/src/main.rs:190`)*

cmd_segments accounts the dead-letter directory by counting EVERY regular file (main.rs:190: `if f.path().is_file()` with no name filter), whereas dl_segments (used by `dl list`/`dl drop`) only matches files that start with `dl_` and end in `.wab.sealed`/`.wab` (main.rs:306-308). So `weir-ctl segments` and `weir-ctl dl list` can report a different dead-letter file count/byte total for the same directory — e.g. a stray `.tmp` partial, an operator note, or any non-`dl_` file inflates the `segments` view but not the `dl list` view. For a 1.0 operator tool, two surfaces describing the same store should agree.

➡️ **Fix:** Reuse dl_segments() (or its predicate) inside cmd_segments' dead_letter branch so both commands apply the identical dl_* / suffix filter.

### G08 — DecodeError doc claims variants are 'ordered by the decode sequence' but the declared order contradicts the actual decode order
*(core · docs/doc · `crates/weir-core/src/error.rs:8`)*

error.rs:7-11 states the variant order is meaningful: 'Variants are ordered by the decode sequence: magic → version → header CRC → payload fields.' But the declared enum order is BadMagic, VersionMismatch, UnknownMessageType, UnknownDurability, HeaderCrcMismatch, PayloadCrcMismatch, TruncatedFrame, PayloadTooLarge (error.rs:15-51). The real decode sequence checks the header CRC (envelope.rs:178-185) BEFORE parsing the typed fields that yield UnknownMessageType/UnknownDurability (envelope.rs:187-190) — so HeaderCrcMismatch fires before those two at runtime yet is declared after them, directly contradicting the doc's own 'header CRC → payload fields' phrasing. Likewise in Envelope::decode, PayloadTooLarge (step 3, envelope.rs:277) fires before TruncatedFrame (step 4, envelope.rs:287) and PayloadCrcMismatch (step 5, envelope.rs:292), yet is declared last. A maintainer trusting the explicit 'ordered by the decode sequence' contract is misled. Distinct from F49/F51 (Fixed) and the 6 escalations.

➡️ **Fix:** Reorder the variants to match the real decode order (BadMagic, VersionMismatch, HeaderCrcMismatch, UnknownMessageType, UnknownDurability, then PayloadTooLarge, TruncatedFrame, PayloadCrcMismatch), or soften the doc to drop the 'ordered by the decode sequence' claim.

### G09 — Payload's PartialEq-against-bytes impls are one-directional (payload == slice compiles, slice == payload does not)
*(core · api/decision · `crates/weir-core/src/payload.rs:99-115`)*

Payload defines `impl PartialEq<[u8]> for Payload` (payload.rs:99), `impl PartialEq<Vec<u8>> for Payload` (payload.rs:105), and `impl PartialEq<&[u8]> for Payload` (payload.rs:111) — all only in the `PartialEq<Other> for Payload` direction. There are no reverse impls, so `payload == some_slice` compiles but `some_slice == payload` does not. The Rust API guidelines (C-COMMON-TRAITS) treat PartialEq as conceptually symmetric and std follows this for analogous cross-type comparisons (e.g. both `String: PartialEq<str>` and `str: PartialEq<String>`). F51 (Fixed, commit b446533) added round-trip/Hash tests but only for the forward direction (tests at payload.rs:165-179), freezing the asymmetry into the 1.0 surface. Adding the reverse impls after 1.0 is non-breaking, so low urgency, but worth a deliberate call before freeze rather than by omission. Genuinely new (F51 was a test-coverage fix, not this asymmetry).

➡️ **Fix:** Either add the three reverse `impl PartialEq<Payload> for X` impls now for symmetry, or document that comparison is Payload-on-the-left only.

### G10 — F05 completeness assert debug_assert_eq!(durable_through, read_index) runs before the read_failed check and is violated by a mid-prefix read error on a resumed segment
*(drain · error-handling/guard · `crates/weir-server/src/drain/mod.rs:698`)*

The F05 refactor (bb7650e) added `debug_assert_eq!(durable_through, read_index);` at drain/mod.rs:698, which runs UNCONDITIONALLY before the `if read_failed` quarantine branch at line 700. The asserted invariant (all readable records durably processed) holds for the read-failed path in general because the trailing non-empty batch is still committed at mod.rs:679-695 — EXCEPT when the read error lands inside the already-skipped prefix of a resumed segment. Reachable case: process_segment is called with `skip = processed > 0` (mod.rs:362-369 / 452-457). On the re-read the reader returns Err while read_index is still <= skip (records 1..skip are `continue`d at mod.rs:654-656 and never entered a batch). The error arm (mod.rs:640-649) sets read_failed=true and breaks WITHOUT incrementing read_index for the failed record. At the assert point durable_through == skip (its initial value, never advanced — no batch committed) but read_index < skip → the assert fails. SegmentReader::next (wab/mod.rs:775-820) genuinely returns Some(Err(_)) on a read_exact failure or a CRC mismatch mid-stream, and a previously-clean prefix record can fail to re-read on a transient I/O fault or post-seal bit-rot (the comment at mod.rs:641 explicitly contemplates "a transient read error"), so the scenario is reachable, if uncommon. In debug/test builds this panics process_segment; run_drain_supervised (mod.rs ~161, MAX_DRAIN_RESPAWNS=10) catches it and respawns, dropping the in-flight segment from the channel for this run (still durable on disk, replayed on restart). In release builds the debug_assert is compiled out and the segment correctly falls through to the read_failed quarantine branch — no incorrect behaviour. Net: a newly-introduced assertion is placed on a path where the invariant is false by construction, crashing debug/test builds on a legitimate mid-prefix read fault during a resume.

➡️ **Fix:** Guard the assert with the non-failed path, e.g. `if !read_failed { debug_assert_eq!(durable_through, read_index); }`, or move it just before the final `ProcessResult::Confirmed { record_count: read_index }` return (after the read_failed early-return), so the completeness check only runs where the invariant actually holds.

### G11 — F05 left the canonical drain state-machine diagram(s) stale: cap-clears no longer goes to Draining/"retry from start"
*(drain · docs/doc · `crates/weir-server/src/drain/mod.rs:17`)*

➡️ **Fix:** Update the mod.rs:17 diagram's last transition to e.g. `BlockedDeadLetterFull ──(cap clears)──▶ RetryingTransient (same segment, RESUMING past already-processed sub-batches)`, matching the inline comments at mod.rs:439-445. ### G12 — F05 post-block retry seeds next_delay=ZERO, defeating exponential backoff into a no-backoff busy-retry loop on subsequent transient sink errors
*(drain · bug/bug-fix · `crates/weir-server/src/drain/mod.rs:452`)*

When a dead-letter-blocked segment gets headroom, the drain now routes the retry through `DrainState::RetryingTransient { next_delay: Duration::ZERO, retries_left: config.max_retries, .. }` (mod.rs:452-457, introduced by F05/bb7650e which replaced the prior `pending.push_front(segment); Draining` path). ZERO is intended only as "retry immediately" (comment mod.rs:443-445). But if that resumed attempt then hits a TRANSIENT sink error (not blocked), the RetryingTransient arm recomputes the next delay as `next_retry_delay(next_delay * 2, retry_after)` = `next_retry_delay(Duration::ZERO * 2, None)` (mod.rs:381). next_retry_delay (mod.rs:559-562) returns `hint.unwrap_or(default).min(MAX)` = ZERO when the default is ZERO and there is no Retry-After hint, so `std::thread::sleep(next_delay)` at mod.rs:352 sleeps 0 again. The segment therefore spin-retries the sink with NO backoff for the whole retries_left budget (default MAX_RETRIES=3). The first-failure path (transition_from_draining, mod.rs:529-534) correctly seeds `next_delay: next_retry_delay(config.base_retry_delay, ..)`, so this is a backoff regression specific to the post-block recovery path — exactly when the sink is under pressure and backoff matters most. A sink-supplied Retry-After hint still overrides ZERO, so only the no-hint transient case spins. Bounded by max_retries, hence low severity, but unintended (the comment documents only the immediate first retry, not a no-backoff loop).

➡️ **Fix:** Seed the post-headroom RetryingTransient with `next_delay: config.base_retry_delay` (matching the Draining path) so the immediate retry still fires (delay can be made 0 only for that first attempt) but subsequent transient failures back off exponentially. Alternatively, have next_retry_delay treat a ZERO default as base_retry_delay.

### G13 — F05 retry/unblock re-reads, re-allocates and re-CRCs every already-processed prefix record before skipping it
*(drain · perf/refactor · `crates/weir-server/src/drain/mod.rs:654`)*

The F05 sub-batch-resume path (process_segment, drain/mod.rs:632-657) resumes a retried or post-block segment by opening a fresh SegmentReader and iterating from record 0, skipping the already-durable prefix with `read_index += 1; if read_index <= skip { continue; }` (mod.rs:651-656). The skip is applied only AFTER SegmentReader::next (wab/mod.rs:766-823) has done the full per-record work: read the 4-byte length and 4-byte CRC, heap-allocate `vec![0u8; payload_len]` (wab/mod.rs:805), read_exact the payload, and compute `crc32fast::hash(&payload_buf)` (wab/mod.rs:811) — all then discarded by the continue. So every transient retry (up to max_retries per attempt) and every BlockedDeadLetterFull→RetryingTransient unblock re-pays a full buffered read + heap allocation + CRC32 over the entire already-processed prefix; under a flapping dead-letter cap the same prefix is re-scanned on each unblock. Correctness is unaffected (the prefix is correctly skipped) but this is avoidable redundant I/O, heap churn and CRC work on the recovery/retry path, in newly-written code (post-65a39fb).

➡️ **Fix:** Add a SegmentReader fast-forward that reads only the 8-byte length+CRC framing and advances the BufReader by payload_len without allocating or CRC'ing skipped records, or persist the byte offset of the resume point so the reader can seek directly to it instead of re-walking from record 0.

### G14 — Metrics HTTP accept loop exits permanently on any transient accept() error
*(worker-queue-metrics · observability/error-handling · `crates/weir-server/src/metrics/server.rs:55`)*

serve()'s accept loop treats EVERY accept() error as fatal: `let Ok((stream, _peer)) = listener.accept().await else { return; };` (server.rs:55-57). tokio's TcpListener::accept surfaces transient, recoverable errors — ECONNABORTED (peer reset between SYN and accept) and EMFILE/ENFILE/ENOBUFS (fd/buffer exhaustion) — directly to the caller without retrying. A single such error permanently kills the metrics endpoint for the rest of the daemon's lifetime, with no log, no metric, and no recovery until restart — most likely to fire precisely under fd pressure, exactly when operators most need metrics. The doc at server.rs:37-38 claims the task exits only on an 'unrecoverable accept error', but the code makes no recoverable/unrecoverable distinction. The data-path accept loop already recognises this class and backs off/retries instead of dying (is_accept_resource_exhaustion in socket/mod.rs:207). This is a separate mechanism from the already-fixed F20 (which added per-handler SCRAPE_IO_TIMEOUT for slowloris/parked handlers); F20 did not touch the accept loop's error handling.

➡️ **Fix:** On accept() Err, log and `continue` (optionally with a brief backoff on resource-exhaustion errors, reusing is_accept_resource_exhaustion from socket/mod.rs) instead of returning, so a transient EMFILE/ECONNABORTED doesn't permanently take down metrics exposition.

### G15 — weir_wab_bytes_on_disk double-counts dead-letter and quarantine bytes (no reserved-subdir skip)
*(worker-queue-metrics · observability/guard · `crates/weir-server/src/main.rs:74`)*

compute_wab_bytes_on_disk (main.rs:58-80, wired to the wab_bytes_on_disk gauge at main.rs:484-488 every 5s) walks every subdirectory of wab_dir and sums any file whose name ends in `.wab` or `.wab.sealed` (line 74). But wab_dir contains two reserved sibling subdirectories that hold non-live-shard data: `dead_letter/` (files named dl_NNNNNNNN.wab.sealed — dead_letter.rs:6,98,104) and `quarantine/` (segments renamed to `<shard>__<orig>.wab.sealed` — recovery.rs:358-377). Recovery deliberately skips BOTH by name (recovery.rs:47: `if name == "quarantine" || name == "dead_letter" { continue; }`), but this metric function has no such guard — it descends into them unconditionally. Result: (1) every dead-letter byte is counted twice, once in weir_dead_letter_bytes_on_disk (the drain's own gauge — metrics/mod.rs:451, updated at drain/mod.rs:428/829/900) and again in weir_wab_bytes_on_disk; and (2) parked quarantine bytes are attributed to the live WAB. The gauge is documented as 'Current total bytes used by WAB segment files on disk' (metrics/mod.rs:382), so an operator alerting on WAB backlog growth gets readings inflated by dead-letter/quarantine backlog — conflating the exact conditions the two gauges exist to separate. (Distinct from the already-fixed F16, which was recovery re-sealing dead_letter as a shard dir; this is the metric-path analogue at a different file:line.)

➡️ **Fix:** In compute_wab_bytes_on_disk, mirror the recovery.rs:47 guard: skip subdirectories named `dead_letter` and `quarantine` before descending, so the gauge reflects only live shard segments.

### G16 — ClientTlsConfig implements no Debug despite holding only Debug-able, non-secret fields
*(client-sdk-ctl · api/refactor · `crates/weir-client/src/tls.rs:14`)*

`ClientTlsConfig<'a>` (tls.rs:14-25) is a public struct whose every field is Debug-able and non-secret (`&Path` x3, `&str`, `Option<Durability>`), yet it derives nothing — no Debug, no Clone. The Rust API guideline C-DEBUG ('all public types implement Debug') flags exactly this, and sibling public config-like types in the workspace do derive Debug (e.g. CommitResult/SinkHealth in weir-sink-sdk/src/lib.rs). Adding the derive is non-breaking later but easy to forget once the surface is locked, and a caller hits it the moment they try to log or assert on a TLS config. No fixed/deferred item covers this type.

➡️ **Fix:** Add `#[derive(Debug, Clone)]` to ClientTlsConfig — the fields are paths/strings and leak no key material (only file paths).

### G18 — Envelope::decode silently ignores trailing bytes past frame_len — undocumented and untested in the wire reference crate
*(core · test/doc · `crates/weir-core/src/envelope.rs:287`)*

Envelope::decode treats `buf.len() < frame_len` as TruncatedFrame (envelope.rs:287) but accepts any `buf.len() >= frame_len`, decoding the first frame and silently discarding everything after byte `frame_len` (= HEADER_LEN + payload_len + 4). This crate is the executable reference for the v1 wire format (lib.rs:18-19), yet the decode doc (envelope.rs:255-265) lists five validation steps and never states what happens to trailing bytes, and no test exercises an over-long buffer. Harmless for weir's own socket/client paths (they read exactly frame_len), but an external implementer comparing against this reference has no spec or test pinning whether 'extra bytes after a complete frame' is rejected or ignored — a latent interop ambiguity to settle at the 1.0 wire freeze. Not covered by the Fixed table or the wire-freeze escalations (F25/F50/F52 concern Nack mapping, payload_len arg, and reserved flags respectively).

➡️ **Fix:** Document the contract on Envelope::decode ('bytes beyond the declared frame length are ignored; the caller owns framing') and add a test asserting a valid frame with appended garbage decodes to the same Envelope — or, if exact-length is intended, return an error for trailing data. Lock the semantics before 1.0.

### G19 — Wire enum TryFrom error structs (UnknownMessageType/UnknownDurability/UnknownNackReason) are not re-exported at the crate root, unlike the enums they belong to
*(core · api/refactor · `crates/weir-core/src/lib.rs:29-34`)*

lib.rs:29-34 re-exports the three frozen wire enums flat at the crate root (`pub use durability::Durability; pub use envelope::{...MessageType}; pub use nack::NackReason;`) and the crate doc (lib.rs:7-16, 'What's here') presents those flat re-exports as THE public API. But each enum's public `TryFrom<u8>` impl uses a separate, public error struct that is NOT re-exported: `UnknownMessageType` (envelope.rs:52, used as the `type Error` at envelope.rs:63), `UnknownDurability` (durability.rs:20, error at durability.rs:31), `UnknownNackReason` (nack.rs:35, error at nack.rs:57). An external caller who writes `weir_core::Durability::try_from(byte)` and wants to name/match/store the resulting error must reach into the undocumented sub-module path `weir_core::durability::UnknownDurability`. These structs are part of the frozen 1.0 surface (they appear in public `TryFrom` signatures), so this is a discoverability/consistency gap that becomes a breaking move-or-rename after 1.0. Confirmed not in the Fixed table nor the 6 escalations.

➡️ **Fix:** Re-export the three error structs from the crate root next to their enums (e.g. `pub use durability::{Durability, UnknownDurability};`), or explicitly document the sub-module path. Free now, a breaking change after 1.0.

### G20 — client_cert_cn parses the full X.509 client certificate (parse + String alloc) on every successful mTLS handshake regardless of log level, purely to populate a debug! field
*(socket · perf/refactor · `crates/weir-server/src/socket/tcp.rs:225`)*

On every successful mTLS handshake the accept task unconditionally binds `let cn = client_cert_cn(&tls_stream);` (tcp.rs:225, verified) before the only consumer, `debug!(client_cn = ?cn, ...)` on the very next line (tcp.rs:226). client_cert_cn runs x509_parser::parse_x509_certificate over the peer-cert DER and allocates a String (tcp.rs:336-346, verified). Because cn is bound eagerly, the parse + allocation happen even at the production-default `info` level where the debug event is never emitted — pure wasted work plus one allocation per connection on the accept hot path, and a small attacker-influenced parse cost (a peer that completes the handshake controls the cert bytes fed to the parser). The function's own doc states it is 'purely diagnostic'.

➡️ **Fix:** Gate the parse behind the active level, e.g. compute cn only inside `if tracing::enabled!(tracing::Level::DEBUG)`, or pass it as a lazily-evaluated tracing field (closure/valuable) so the parse runs only when the debug event is actually recorded.

## ⚪ Refuted / already-known

- **Graceful-shutdown drain budget (shutdown_timeout_secs) is never validated against the hardcoded 30s ACK_TIMEOUT, so the documented drain precondition is silently unmet — in-flight handlers can be abort_all'd before their ack/nack completes** (socket) — All cited facts verified at HEAD. ACK_TIMEOUT is a hardcoded 30s (connection.rs:35). The handle_push ack wait is capped by exactly that timeout (connection.rs:362, ack_timeout==ACK_TIMEOUT). shutdown_timeout_secs defaults to 30 and is validated ONLY against ==0 (config/mod.rs:519-525) — no relation to ACK_TIMEOUT is enforced. Both drain loops document the precondition "timeout should be >= ACK_TIMEOUT + buffer" (mod.rs:231-233, tcp.rs:267-268) yet nothing checks it, and abort_all fires at mod.rs:249 / tcp.rs:283 with no startup diagnostic — only the runtime counter connections_aborted_at_shutdown (metrics/mod.rs:182-187). So the default leaves zero buffer and an operator can set shutdown_timeout_secs=1, and the documented graceful-drain contract is genuinely unenforced/unwarned. Crown invariant is NOT violated (no false ack; at-least-once + abort_all backstop), as the finding itself states. The proposed fix is low-risk and already has a sibling precedent: config/mod.rs:403-409 pushes a post-validation advisory warning for a too-small dead_letter_max_bytes, structurally identical to the suggested shutdown_timeout warning.
