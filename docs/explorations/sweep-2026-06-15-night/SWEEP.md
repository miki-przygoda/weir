### S01 — Drain confirms+deletes a sealed segment on a post-seal tail truncation: lost trailing records read as a clean end-of-stream (silent false confirm / data loss)
- **Track / agent:** inv-confirm-delivered  ·  **fix-risk:** moderate
- **Location:** `crates/weir-server/src/drain/mod.rs:744-748 (confirm decision) + crates/weir-server/src/wab/mod.rs:801-808 (SegmentReader::next clean-EOF)`
- **Claim:** SegmentReader::next() treats UnexpectedEof on a record length field as a CLEAN end-of-stream (wab/mod.rs:803 `Err(e) if e.kind()==UnexpectedEof => return None`), identical to hitting the zero-length sentinel (wab/mod.rs:811-813). process_segment relies solely on that None to end its read loop, then asserts `durable_through == read_index` and returns `ProcessResult::Confirmed { record_count: read_index }` (drain/mod.rs:744-748) using the number of records IT MANAGED TO READ — never the authoritative count. The segment footer carries the true `record_count`/`data_bytes`/`file_crc32` (format.rs:21-24) and `read_segment_record_count` already reads it at replay time (wab/mod.rs:343), but ONLY to feed the `recovery_records_replayed` metric (wab/mod.rs:351); nothing ever compares the footer's record_count (or file_crc32) against what the drain actually delivered. Consequently a sealed segment whose TAIL is lost after seal — trailing complete records plus the sentinel+footer truncated away by partial media loss / a torn block on a non-atomic fs after a power cut / an at-rest truncation while the segment waits on disk during a long sink outage or BlockedDeadLetterFull episode — is read as a shorter, structurally-clean stream with read_failed=false, so it is confirmed and DELETED (next_state_after_process -> confirm_and_delete, drain/mod.rs:511-512). The dropped tail records were acked durable to producers but never delivered to the sink or dead-lettered: a silent crown-invariant violation. This evades the two existing defenses — per-record CRC mismatch and mid-stream read errors both return Some(Err) and trigger quarantine (drain/mod.rs:655-663, process_segment:718-735), but a tail truncation produces no error at all. It is distinct from recovery's truncation handling (recovery.rs only rebuilds UNSEALED .wab files and writes a fresh footer; an already-.sealed file goes straight to the drain) and is not covered by the DST harness, which injects faults only at write/fsync/seal/rename time during segment CREATION (dst.rs Fault variants), never at-rest corruption of an already-sealed file the drain reads. No prior F*/G* item covers the drain-read-side completeness check.
- **Fix:** On the drain read path, validate completeness against the footer before confirming: read the footer's record_count (reuse read_segment_record_count) when opening the segment and, after the read loop ends WITHOUT a read error, require read_index == footer.record_count; on mismatch treat the segment as corrupt (quarantine via wab::recovery::quarantine and return ProcessResult::Quarantined) rather than confirming. Optionally also verify the footer's file_crc32 over the bytes before the sentinel for full at-rest integrity. A lighter-weight variant: have SegmentReader detect a missing sentinel/footer (it currently cannot distinguish 'reached sentinel' from 'file just ended') and surface a TruncatedTail error so the existing read_failed quarantine branch fires. Note the trade-off: this converts a previously-undetected silent loss into an operator-visible quarantine, which is the correct end state for the durability invariant.
- **Tests:** Add a drain unit test mirroring recovery_crash_simulation_truncate_mid_record but on a SEALED segment: seal a 3-record segment, then set_len() it to drop the 3rd record and the sentinel/footer, send it to the drain, and assert the segment is NOT confirmed+deleted (no .confirmed sidecar, original preserved/quarantined) and that records_committed reflects only the 2 survivors are NOT silently treated as the whole segment. Add a second test truncating exactly at the sentinel boundary (footer gone, sentinel present) to pin behaviour. If footer-CRC validation is added, add a test that flips a payload byte after seal and asserts quarantine rather than confirm.
- **Verified:** Independently reproduced: SegmentReader::next() maps UnexpectedEof on a length field to a clean None (wab/mod.rs:803), indistinguishable from the sentinel; process_segment ends on None and returns Confirmed{record_count: read_index} (drain/mod.rs:746-748) using what it read, never the footer's authoritative record_count/file_crc32 (read only for a metric at wab/mod.rs:351). The debug_assert_eq (744) compares two read-derived values and passes on a truncated tail (and is compiled out in release). Confirmed -> confirm_and_delete (511-512) deletes the segment, so a post-seal tail truncation silently drops records acked durable but never delivered/dead-lettered. Recovery leaves sealed files untouched (recovery.rs:19) and DST only injects write/seal/rename-time faults (dst.rs), so neither covers at-rest sealed-file truncation; no prior F*/G* item covers a drain-read completeness check (G10 is only about assert ordering).

### S02 — Response writes (Ack/Nack/HealthCheckResponse) on the data-path connection have no write timeout — a slow/non-reading client pins a connection (and its semaphore permit) forever
- **Track / agent:** sec-wire  ·  **fix-risk:** moderate
- **Location:** `crates/weir-server/src/socket/connection.rs:465 (send_nack), :493 (send_ack), :281 (healthcheck_response write); permit lifetime: crates/weir-server/src/socket/mod.rs:193-202 and crates/weir-server/src/socket/tcp.rs:175-243`
- **Claim:** Every server->client write on the data path is a bare `stream.write_all(...).await` with NO timeout and NO race against shutdown. The read side is fully hardened — each read_exact is wrapped in `tokio::time::timeout(config.read_timeout, ...)` AND raced against `shutdown_rx` (connection.rs:110-117, :203-207, :226-230) — but the matching write side is not. A hostile producer can complete one or more valid frames and then stop reading: the kernel send buffer fills and `write_all` blocks indefinitely (TCP+mTLS and Unix alike; rustls buffering only delays the stall, it does not bound it). Because the connection task owns its `_permit` for the entire `handle_connection` future, a wedged write holds a `max_connections` semaphore permit forever. With the default `max_connections = 256`, ~256 cheap stalled connections exhaust the global cap (shared across Unix + TCP) and block ALL legitimate producers from connecting — a sustained ingest denial of service at trivial attacker cost. The same wedged tasks also never complete `join_next()`, so graceful shutdown cannot drain them and is forced into `abort_all` after `shutdown_timeout_secs` (mod.rs:235-251 / tcp.rs:277-292). This is a real, unguarded hole and is NOT covered by prior findings: F20 hardened only the metrics endpoint; F26/F27/F29 hardened only reads/TLS-handshake against stalls. Critically, the metrics server already established the exact precedent for the fix and the exact threat model — server.rs:118-124 bounds its response writes with `SCRAPE_IO_TIMEOUT` because 'a peer that stops reading mid-response must not pin the handler/permit' — yet the higher-value ingest path lacks the equivalent guard. The existing test `stalled_client_does_not_block_other_connections` (tests/system.rs:1099) masks the gap: it stalls a SINGLE connection with a 5-byte payload whose 20-byte Ack fits entirely in the socket buffer, so write_all completes and the task suspends on the next read — it never exercises a write that actually blocks (which requires pipelining enough frames that the backed-up Acks exceed the kernel send buffer, or filling the cap with many such connections).
- **Fix:** Bound every data-path response write the same way the metrics server does: wrap send_ack/send_nack and the healthcheck-response write in `tokio::time::timeout(config.read_timeout, write)` (reuse the existing per-connection read_timeout as the write deadline — it is already the slow-peer bound), and on Elapsed bump `connection_idle_timeout` (or a new write-timeout counter) and return Ok(()) to drop the connection and release the permit. Optionally also race the write against `shutdown_rx.changed()` so a wedged writer releases promptly during graceful shutdown (mirrors the read-side shutdown race added in F26). A bounded write deadline is sufficient and the lower-risk choice; SO_SNDTIMEO/TCP_USER_TIMEOUT are not needed.
- **Tests:** 1) Unit test in connection.rs over a tokio::io::duplex pipe with a tiny write-buffer capacity (or a custom AsyncWrite that never drains): pipeline enough Push frames that the accumulated Acks block write_all, assert handle_connection returns within ~read_timeout and increments the timeout counter rather than hanging. 2) Extend tests/system.rs stalled-client test to a 'pipelining non-reader' variant: open a connection, send several thousand Push frames (or large enough to overflow the send buffer), never read, and assert the daemon stays responsive to a fresh client and that the stalled connection is dropped within read_timeout. 3) A shutdown test: a connection wedged on a write must be released by the shutdown signal without waiting out read_timeout (parallel to shutdown_releases_connection_stalled_mid_payload).
- **Verified:** Verified against current code: send_ack (connection.rs:493), send_nack (:465), and the HealthCheckResponse write (:281) are bare write_all with no timeout and no shutdown race, while both reads ARE wrapped in timeout+shutdown_rx (:110-117,:203-207); _permit is held for the whole handle_connection future (mod.rs:193-202) against a single global default-256 semaphore (config/mod.rs:478, main.rs:510); no SO_SNDTIMEO/TCP_USER_TIMEOUT exists; metrics server already bounds writes with SCRAPE_IO_TIMEOUT for this exact threat (server.rs:118-124); F20/F26/F27/F29 and SCAN.md do not cover the data-path write; the cited test (system.rs:1099) sends one 5-byte frame whose 20-byte Ack fits the send buffer, so write_all returns and it never exercises a blocking write.

### S03 — Drain CommitResult partition-violation guard (F02 false-ack defense) has no test
- **Track / agent:** test-coverage  ·  **fix-risk:** safe
- **Location:** `crates/weir-server/src/drain/mod.rs:800-810 (commit_batch: `accounted != payloads.len()` guard)`
- **Claim:** commit_batch enforces the Sink contract that committed ∪ dead_lettered must partition the input batch: if `committed.len() + dead_lettered.len() != payloads.len()` it logs an error and returns BatchResult::Transient to preserve the segment rather than confirm+delete it. This is the F02 fix — without it, a (non-conforming, third-party) sink that drops a record from BOTH vectors would have its segment confirmed-and-deleted with that record neither delivered nor dead-lettered: a silent false ack with zero detection, violating the crown invariant. No test exercises this branch: searching drain/mod.rs for `accounted`/`does not account`/`contract violation` finds only the production code, and every MockSink response in the suite partitions correctly. The guard is therefore unprotected against a future refactor that drops or weakens it. The test is trivially feasible — MockSink::with_responses accepts an arbitrary CommitResult, and commit_batch is already called directly in unit tests (e.g. partial_dead_letter_with_failed_write_is_transient_not_ok at line 1533) — so an under-accounting response can be injected through the public CommitResult::new(committed, dead_lettered).
- **Fix:** Add a unit test `under_accounting_commit_result_is_transient_not_ok` in drain/mod.rs::tests: build a 2-record payloads vec; drive a MockSink whose single response is `Ok(CommitResult::new(vec![Payload::from(b"only-one".as_ref())], vec![]))` (accounts for 1 of 2); call commit_batch directly (via block_on) and assert the result is BatchResult::Transient (segment preserved, NOT Ok). Optionally add a second case where both vectors are empty for a non-empty batch. Assert the segment is not confirmed and that records_committed is not bumped for the dropped record.
- **Tests:** under_accounting_commit_result_is_transient_not_ok (crates/weir-server/src/drain/mod.rs::tests, calling commit_batch directly with a MockSink injecting an under-accounting CommitResult)
- **Verified:** Confirmed: the under-accounting guard exists at drain/mod.rs:800-810 (F02 defense), and a full-repo search finds zero tests exercising the `accounted != payloads.len()` branch — every MockSink response partitions correctly; the proposed test is feasible via public CommitResult::new + with_responses + direct block_on(commit_batch) (template at line 1533).

### S05 — Decreasing shard_count across a restart orphans sealed-but-undrained segments forever (replay scans 0..shard_count, recovery scans all dirs)
- **Track / agent:** inv-crash-recovery  ·  **fix-risk:** moderate
- **Location:** `crates/weir-server/src/wab/mod.rs:316 (replay_unconfirmed) vs crates/weir-server/src/wab/recovery.rs:27-52 (recover_open_segments); reachable via crates/weir-server/src/config/mod.rs:444-445 (shard_count freely 1..256, no on-disk check) and crates/weir-server/src/main.rs:442`
- **Claim:** recover_open_segments iterates every shard directory found under wab_dir (recovery.rs:30 read_dir(wab_dir)), so on startup it seals active .wab files in ALL shard_NN dirs — including dirs whose index is >= the configured shard_count. But replay_unconfirmed only loops `for shard_id in 0..shard_count` (mod.rs:316) and therefore only queues sealed segments from shard_00..shard_{count-1} to the drain. shard_count is operator-configurable in 1..=256 with NO validation against the number of shard directories already on disk (config/mod.rs:444-445). Consequently, if an operator reduces shard_count across a restart (e.g. 8 -> 4 on a k8s resize), every seg_*.wab.sealed in shard_04..shard_07 holding records that were acked durable to producers but never delivered to the sink becomes permanently invisible to the drain: they are never replayed, never confirmed, never dead-lettered, and produce no error or warning. This is a silent, permanent no-delivery of acknowledged records — a violation of the no-loss-across-restart invariant. It is documented as a known hard problem in docs/explorations/phase4-k8s-operator.md:192-214 (which explicitly notes 'any fix would need to iterate all existing shard directories'), but it is NOT fixed in code and there is no runtime guard or warning. main.rs:59 only reads wab_dir for a bytes-on-disk metric; nothing detects orphaned shard dirs. Distinct from all F*/G* items.
- **Fix:** At startup, enumerate the on-disk shard directories and refuse to start (or loudly warn + halt) if any shard_NN dir with NN >= shard_count contains a .wab/.wab.sealed/.wab.confirmed file — surfacing the orphaned-backlog condition before any record is acked. Cheaper interim: have replay_unconfirmed (and recovery) iterate ALL shard dirs present under wab_dir rather than 0..shard_count, queuing every sealed-but-unconfirmed segment regardless of the configured count, so a reduced shard_count cannot orphan acked records. At minimum, emit an error-level startup log + a metric when extra shard dirs are detected.
- **Tests:** Add an integration test: create shard_00..shard_03 each with a sealed-but-unconfirmed segment, run replay_unconfirmed with shard_count=2, and assert the drain channel received segments from shard_02/shard_03 too (currently it would not). Add a config/startup test asserting startup refuses or warns when on-disk shard dirs exceed configured shard_count with non-empty backlog.
- **Verified:** Independently reproduced: replay_unconfirmed loops 0..shard_count (wab/mod.rs:316) while recover_open_segments scans all dirs (recovery.rs:30); shard_count has no on-disk validation (config/mod.rs:445) and main.rs:442 passes it directly, so reducing shard_count across a restart leaves sealed-but-unconfirmed segments in shard_NN (NN>=count) never queued, never confirmed, with no guard/warning — confirmed as a known-unfixed problem in phase4-k8s-operator.md and absent from prior sweep/scan passes.

### S09 — TCP+mTLS listener task is fire-and-forget — drop(rt) cancels its graceful drain as soon as the Unix accept loop returns, defeating shutdown_timeout_secs for the TCP transport
- **Track / agent:** concurrency  ·  **fix-risk:** moderate
- **Location:** `crates/weir-server/src/main.rs:574-590 (tokio::spawn of tcp::run, result discarded) and main.rs:637 + main.rs:659 (Unix socket::run is the only future awaited by block_on; drop(rt) follows)`
- **Claim:** The TCP+mTLS accept loop is launched with a bare `tokio::spawn(async move { tcp::run(...).await })` whose JoinHandle is dropped (main.rs:574-590). The only future the runtime actually waits on is the Unix `socket::run(...).await` at main.rs:637; `rt.block_on(...)` returns the instant that Unix future resolves, and the very next statement is `drop(rt)` (main.rs:659). Tokio's `block_on` returns when the passed future completes — it does NOT wait for other spawned tasks — and dropping the runtime cancels every still-pending task at its current await point. So the shutdown interleaving is: signal fires -> both `shutdown_tx` and `tcp_shutdown_tx` are sent (main.rs:618-621) -> the Unix loop and the TCP `run` task begin draining their JoinSets concurrently -> whichever Unix `drain_join_set` finishes first ends `block_on` -> `drop(rt)` aborts the TCP `run` task wherever it is, including mid-`drain_join_set` (tcp.rs:277) and including the spawned per-connection TLS handler tasks. The TCP loop's own graceful-drain budget (`shutdown_timeout_secs`, tcp.rs:276-292) is therefore never honoured end-to-end: it is silently truncated to 'however long the Unix drain happened to take.' In the common production case where the Unix socket has zero in-flight connections at shutdown, the Unix `drain_join_set` completes in milliseconds, so active TCP connections are hard-aborted almost immediately instead of being given up to `shutdown_timeout_secs` to finish their in-flight push. Operationally this is an asymmetry between transports: a TCP/mTLS client mid-push at SIGTERM has its connection cut and receives no ack/nack (it must retry on reconnect), whereas an identical Unix client is drained cleanly. The crown invariant is NOT violated — a WorkUnit already pushed to the crossbeam queue is still drained and durably written by the workers/flushers after `drop(rt)`, so this is at-least-once (a possible missing-ack/duplicate, never a false ack/data loss). The bug is that the documented TCP graceful-drain contract (tcp.rs module docs: 'Returns ... after draining in-flight connections within shutdown_timeout_secs') is unmet, and `connections_aborted_at_shutdown` may not even be incremented for the TCP side because the abort happens via runtime drop rather than the loop's own `abort_all` path (tcp.rs:288-290).

Concretely: have the `#[cfg(feature = "tls")]` block return `Some((tcp_shutdown_tx, tcp_join_handle))` instead of just the shutdown sender; after the Unix `socket::run(...).await?` returns inside `block_on`, `if let Some(h) = tcp_join_handle { let _ = h.await; }` so both transports finish draining before `block_on` returns and `drop(rt)` runs. (The signal task already fires `tcp_shutdown_tx` so the TCP loop will break promptly; this change only ensures the subsequent drain is actually awaited.) Alternatively, drive both `socket::run` and `tcp::run` from the same `tokio::join!` inside `block_on`.
- **Tests:** Add an integration test in crates/weir-server/tests/tls_listener.rs that: starts the daemon with the TCP+mTLS listener AND no active Unix connections, opens a TCP/mTLS connection and begins a Push whose ack is artificially delayed (e.g. a slow/blocked sink so the flusher hasn't acked yet), sends SIGTERM, and asserts the client either receives its ack/nack OR the connection stays open until ~shutdown_timeout_secs rather than being severed within a few ms. A unit-level test can also assert that main wires a JoinHandle await for the TCP task (e.g. by refactoring the spawn into a small helper returning the handle and asserting the handle is awaited before runtime drop).
